# WebForge v0.2 Harness 设计

> **状态：** 已在对话中确认，现写入仓库供审阅
> **日期：** 2026-03-30
> **范围：** 将 WebForge 重构为符合 `learn-claude-code` 思路的 agent-first harness，并把 `superpowers` 集成为工作流增强层，而不是主控制面。

---

## 1. 目标

WebForge v0.2 应该成为一份可实际使用的 `harness engineering` 样板与基础设施：

- 遵循 `learn-claude-code` 的核心思想：由模型负责决策，由 harness 提供环境、状态、工具、观察面和恢复机制。
- 同时支持 `Codex` 与 `Claude Code`，而不是为不同 agent 维护两套真相。
- 深度集成 `superpowers skills`，但不能让 `superpowers` 变成唯一记忆源、唯一调度器或唯一规则层。
- 先把 `work-forge` 自身打造成高质量样板仓库，再从中抽象出可复用的方法论。

这一版是一次明确的**重构重建**，不是在现有 `.webforge` 结构上做兼容性补丁。

## 2. 非目标

v0.2 不追求一步完成所有高级能力。

以下内容不在本版本的完整实现范围内：

- 完整成熟的 worktree 编排能力
- 完整的权限治理引擎
- 超出最小持久化协议之外的分布式多 agent runtime
- 让 harness 依赖单一 agent 平台的深度 provider 绑定

这些能力可以在 v0.2 契约稳定后继续迭代。

## 3. 设计驱动因素

### 3.1 第一性原理

harness 必须体现以下原则：

1. **Agent-first**  
   仓库必须假设 agent 可以通过读取契约和状态继续工作，而不是依赖 CLI 代替 agent 思考。

2. **状态落盘**  
   项目的当前真相必须保存在 `.webforge/`，而不是停留在聊天上下文中。

3. **单一契约，多种 agent 环境**  
   `Codex` 和 `Claude Code` 必须能够面向同一份仓库契约工作。

4. **增强工作流，而不是接管工作流**  
   `superpowers` 用来增强工作方法，而不是替代核心状态模型。

5. **恢复优先于记忆**  
   新 session 应该通过读取文件恢复，而不是依赖先前会话记忆。

### 3.2 为什么 v0.1 不够

当前仓库虽然已经受到了正确理念的影响，但实现上仍然更接近 `CLI-first orchestration`，而不是 `agent-first harness`。

主要问题包括：

- 命令处理器承担了过多决策逻辑。
- 当前的 `Agent` 抽象更像交付物生成器，而不是真正面向 runtime 的 agent loop。
- session 恢复还不足以成为可靠的连续性机制。
- 文档与实现尚未在“由谁驱动工作流”这个问题上达成一致。

v0.2 的目的就是补上这层差距。

## 4. 目标模型

WebForge v0.2 应当由五层组成。

### 4.1 Contract Layer

这是面向仓库的约定层，定义任何 agent 进入仓库后必须遵守什么。

主要产物：

- `AGENTS.md`
- `README.md`
- `ARCHITECTURE.md`
- `docs/methodology/*.md`

职责：

- 定义 WebForge 的心智模型
- 定义启动与恢复顺序
- 定义 agent 如何读取和更新项目状态
- 定义哪些内容属于硬规则、哪些属于指导或增强

### 4.2 State Layer

这是项目活动状态的唯一持久化真相源。

主要产物：

- `.webforge/config.yaml`
- `.webforge/tasks.json`
- `.webforge/phases.json`
- `.webforge/runtime.json`
- `.webforge/knowledge/*`
- `.webforge/deliverables/*`
- `.webforge/mailboxes/*`
- `.webforge/sessions/*`
- `.webforge/logs/*`

职责：

- 持久化项目状态
- 持久化任务图和依赖状态
- 持久化恢复上下文
- 持久化通信记录
- 持久化交付物索引

### 4.3 Runtime Layer

这是实际的 harness 核心。它不能变成第二份政策文档，也不能假装自己是 agent 大脑。

主要模块：

- `workspace`
- `context`
- `planning`
- `runtime`
- `task`
- `session`
- `mailbox`
- `deliverable`

职责：

- 加载并校验 workspace
- 从持久化状态构建 agent 执行上下文
- 安全执行 ready-task loop
- 持久化任务、session 和交付物状态流转
- 对 CLI 和 agent 入口暴露稳定内部 API

### 4.4 Adapter Layer

这一层用于处理平台差异，但不能改变项目真相模型。

主要关注点：

- `Codex` 的使用方式
- `Claude Code` 的使用方式
- skill 调用方式差异
- session 入口差异

职责：

- 在平台行为和 WebForge 契约之间做薄适配
- 避免把平台特有假设泄漏进 `.webforge/`

### 4.5 Workflow Enhancement Layer

`superpowers` 应该放在这一层。

主要使用场景：

- `brainstorming`
- `writing-plans`
- `strategic-compact`
- `continuous-learning-v2`
- 领域实现类 skills

职责：

- 提高设计、计划、review 和学习质量
- 为特定工作类型路由更合适的方法
- 绝不替代仓库契约或状态模型

## 5. WebForge 与 Superpowers 的关系

这层关系必须明确：

- `WebForge` 定义项目如何运作。
- `.webforge` 定义当前事实状态是什么。
- `Codex` 与 `Claude Code` 面向这份事实状态执行。
- `superpowers` 定义特定工作类型该如何做得更好。

这意味着：

- `superpowers` 不是唯一记忆层
- `superpowers` 不是唯一计划层
- `superpowers` 不是任务状态数据库
- `superpowers` 不是 runtime 调度器

它是**面向 agent 的工作流引擎**，而不是项目的持久化操作系统。

## 6. 兼容性策略

WebForge v0.2 的设计目标是：

- `Codex + Claude Code` 双兼容
- 深度但有边界的 `superpowers` 集成

这意味着：

- 一份仓库契约
- 一套 `.webforge` 状态模型
- 一套 runtime 核心
- 薄的平台适配层

它明确拒绝：

- 为不同 agent 维护平行真相
- 把主项目状态写进 provider 专属记忆系统
- 允许 CLI 命令文件成为真正的 orchestrator

## 7. 仓库契约

### 7.1 必需的启动顺序

任何兼容 agent 都应该通过以下顺序恢复工作：

1. 读取 `AGENTS.md`
2. 读取 `.webforge/runtime.json`
3. 读取 `.webforge/tasks.json`
4. 读取 `.webforge/phases.json`
5. 从 `.webforge/sessions/index.json` 读取 session 元数据
6. 视情况检查未处理邮箱消息
7. 从 knowledge、deliverables 与任务状态构建当前上下文
8. 决定下一个 ready task 或当前阻塞状态

### 7.2 必需的回写顺序

当工作引发项目状态变化时，agent 或 runtime 必须回写 `.webforge/`。

例如：

- 新任务图 -> `tasks.json`、`phases.json`
- 新运行时状态 -> `runtime.json`
- 新交付物 -> `deliverables/index.json`
- 新 session 快照 -> `sessions/index.json` 与对应 session 文件
- worker 通信 -> `mailboxes/*.jsonl`

### 7.3 硬规则与指导的区别

仓库契约应该明确区分：

- **硬规则**  
  任何兼容 agent 都必须遵守的规则

- **运行时协议**  
  状态流转和持久化规则

- **工作流指导**  
  关于 `superpowers` 或平台最佳实践的建议

这种分离可以避免规则、状态和习惯混在一起。

## 8. v0.2 Workspace 契约

v0.2 明确**不保留**当前 `.webforge` 结构中与新模型冲突的部分。

目标 workspace 结构：

```text
.webforge/
  config.yaml
  tasks.json
  phases.json
  runtime.json
  knowledge/
    index.json
    requirements/
    design/
    decisions/
    parsed/
  deliverables/
    index.json
  mailboxes/
  sessions/
    index.json
  logs/
```

### 8.1 `runtime.json`

这个文件应该成为恢复流程的最小机器可读入口。

它应当包含：

- workspace 版本
- 当前 session id
- 当前 phase id
- 当前 task id
- runtime 最后更新时间
- 当前模式或状态摘要

### 8.2 `tasks.json`

它仍然是持久化任务图。

必须清晰支持：

- 状态
- 依赖边
- assignee 或角色意图
- 上下文与 review 要求所需的 metadata

### 8.3 `sessions/`

session 恢复必须可持久化且可审计。

session 模型应支持：

- 最新 session 查找
- 当前 task 与 phase 快照
- 当时正在做什么的摘要
- 面向新 agent session 的 resume 提示

### 8.4 `deliverables/index.json`

交付物不能只是一堆散文件，必须有索引。

索引应支持：

- deliverable id
- 来源任务
- 类型
- 路径
- review 状态
- 创建时间

## 9. Runtime 设计

### 9.1 核心模块

规划中的 runtime 模块：

```text
src/core/
  workspace.ts
  context.ts
  planning.ts
  runtime.ts
  session.ts
  task.ts
  mailbox.ts
  deliverable.ts
```

### 9.2 模块职责

#### `workspace.ts`

- 校验 `.webforge/`
- 初始化缺失的契约文件
- 暴露规范化路径
- 报告版本或 schema 不匹配

#### `context.ts`

- 汇总当前任务上下文
- 组合 knowledge、deliverables、state 与 runtime 元数据
- 构建提供给 agent runtime 的执行视图

#### `planning.ts`

- 从项目 knowledge 推导或刷新任务图
- 保持计划逻辑不再放在 CLI 命令文件里
- 将 planning 结果持久化回 workspace 状态

#### `runtime.ts`

- 实现 ready-task loop
- 认领或选择可运行任务
- 调用 agent 执行路径
- 持久化状态流转与执行摘要
- 记录失败与 blocked 状态

#### `session.ts`

- 启动 session
- 维护 heartbeat
- 持久化快照
- 查找最新 session
- 生成 resume guidance

#### `task.ts`

- 管理任务状态流转
- 强制依赖解锁规则
- 跟踪 phase 进度

#### `mailbox.ts`

- 持久化 worker 间消息
- 支持未读和过滤查询
- 支持工作流通知

#### `deliverable.ts`

- 创建交付物条目
- 维护交付物索引
- 支持 pending review 状态

## 10. Agent Runtime 方向

面向 runtime 的 `Agent` 抽象不应再假装自己就是整个系统。

它应收缩成一个窄门面：

- 接收标准化任务上下文
- 执行一个边界清晰的工作单元
- 返回结构化输出
- 让 runtime 拥有持久化与流程编排权

这意味着未来应朝着类似这样的结构拆分：

```text
src/agent/
  index.ts
  context.ts
  handlers/
```

关键变化在于：`Agent` 不应再承担杂糅的 workspace 读取、伪编排和内容模板化逻辑。

## 11. CLI 在 v0.2 中的角色

CLI 命令可以保留，但只能作为兼容性与便利入口。

### 11.1 CLI 允许承担的职责

- 解析参数
- 调用 runtime 服务
- 为人类渲染摘要
- 初始化 workspace

### 11.2 CLI 禁止承担的职责

- 拥有真正的任务计划逻辑
- 成为主调度器
- 成为规范恢复系统
- 变成事实上的 agent 大脑

### 11.3 目标命令

- `init`  
  创建 v0.2 workspace 结构

- `plan`  
  调用 planning core 构建或刷新任务图

- `run`  
  调用 runtime 的 ready-task 执行循环

- `resume`  
  恢复最新持久化 session，并输出 next-action guidance

- `dashboard`  
  只做状态观察

- `knowledge` 与 `mailbox`  
  作为辅助读写和观察工具

## 12. 方法论文档

仓库应明确记录 harness 背后的方法论。

计划新增文件：

```text
docs/methodology/
  harness-principles.md
  codex-claude-compat.md
  superpowers-integration.md
  session-recovery.md
```

这些文档应解释：

- WebForge 如何定义 harness
- 哪些东西属于仓库契约、runtime、增强层
- 如何让同一仓库同时被 `Codex` 和 `Claude Code` 使用
- 如何使用 `superpowers` 而不让它变成隐藏依赖

## 13. 迁移立场

这一版不应被当前 `.webforge` 结构绑架。

实际立场是：

- 按 v0.2 模型重新定义 workspace
- 围绕这套模型重建 runtime 模块
- 后续可选再补 reset 或 migration 路径

但 v0.2 的设计决策应优先保证正确性，而不是优先保留早期偶然形成的结构。

## 14. v0.2 交付范围

本版本确认的交付范围为：

1. 重写面向仓库的契约文档
2. 定义并创建新的 `.webforge` 契约
3. 引入 runtime 核心模块
4. 将 CLI 缩减为兼容层
5. 建立 `Codex` 与 `Claude Code` 双兼容
6. 将 `superpowers` 作为有边界的增强层接入

## 15. 验收标准

如果满足以下全部条件，就说明 WebForge v0.2 成功达标。

### 15.1 契约与实现一致

文档和代码必须描述同一个系统。

不能再出现以下矛盾：

- 文档说 WebForge 是 agent-first
- 实现却还是 CLI-first

### 15.2 持久化恢复成立

一个新 session 必须能够仅凭以下内容恢复项目状态：

- `AGENTS.md`
- `.webforge/runtime.json`
- task 与 phase 文件
- session 元数据

而不依赖历史聊天上下文。

### 15.3 Runtime 拥有主循环

`init -> plan -> run -> resume` 必须能够走通，但真正的执行主路径必须位于 runtime 模块，而不是命令处理器里。

### 15.4 最小协作闭环为真

以下四件事必须真实串联起来：

- tasks
- sessions
- deliverables
- mailboxes

不能只停留在“目录已经建出来”的承诺层面。

### 15.5 双兼容成立

`Codex` 和 `Claude Code` 都必须能通过同一份契约和状态模型操作仓库。

### 15.6 Superpowers 集成方式正确

`superpowers` 必须清晰有价值，但始终只是：

- 工作流增强层
- 不是唯一控制面
- 不是唯一记忆机制
- 不是持久化项目 runtime

## 16. 建议的下一步

一旦这份 spec 在仓库内被确认，下一步就是为 v0.2 重构编写实现计划。

这个计划应重点覆盖：

- 契约文档重写
- workspace schema 重设计
- runtime 核心抽取
- CLI 兼容层收缩
- recovery 与 session 稳定化

只有在书面 spec 被接受之后，才进入实现阶段。
