# WebForge v0.1 内部工程进展

日期：`2026-03-31`

这份文档不面向对外发布，而是面向内部工程同步。

目标只有一个：明确当前这套 WebForge 已经做到哪里、现在可用来干什么、还差什么、下一阶段最值的是什么。

## 1. 目标

当前里程碑要解决的问题是：

- 把 WebForge 从“受 `learn-claude-code` 启发的想法”收敛成一套真正可运行的 repo-side harness
- 让 `Codex / Claude Code` 直接在仓库里工作，而不是再包一层伪 orchestrator
- 把 `superpowers` 放在正确位置：方法增强层，而不是状态真相源
- 让 onboarding、recovery、observation、drift、thread、compact 和 workflow artifact 形成可恢复闭环

## 2. 当前已完成

### 2.1 Repo-side harness contract

已完成：

- 以 `.webforge/` 作为仓库内事实源
- 初始化时生成 `AGENTS.md`
- 初始化时生成 `docs/agent-guide.md`
- 初始化时生成 `docs/methodology/superpowers-integration.md`
- 初始化时生成 `docs/examples/agent-onboarding-protocol.md`

结论：

新仓库初始化后，已经不是“只有 CLI 没有 contract”的空壳，而是自带 onboarding 协议的 harness 骨架。

### 2.2 Onboarding / recovery

已完成：

- `doctor --json`
- `resume --json`
- `logs runtime --json`
- `onboard --json`
- `verify init`

当前特点：

- `onboard` 已经是统一入口
- `resume` 已能输出 `nextAction / shouldRead / runtimeLog / threadLinkage`
- `logs runtime --json` 已能同时给出日志对应恢复快照和当前工作区恢复快照
- `doctor --json` 已能显式报告 `runtime-context-drift`

### 2.3 Observation / drift

已完成：

- runtime observation 事件落盘
- dashboard 可读最近 runtime 观察
- `logs runtime` 可读完整事件流
- `contextDrift` 已贯通到 `resume / onboard / doctor`

结论：

当前不是“只能知道出错了”，而是已经能知道“哪条恢复链发生了漂移”。

### 2.4 Superpowers integration

已完成：

- `superpowers` capability / run 持久化
- `webforge superpowers record ...`
- `workflowContext` 贯通 task / runtime / session
- `threadLinkage` 落到 `.webforge/threads.json`
- compact 来源 session 可检查、可恢复判断

结论：

`superpowers` 现在不是旁路说明文档，而是已经能把方法层结果变成仓库内恢复线索。

### 2.5 Verification / smoke / coverage

已完成：

- `init` 自带 post-init self-check
- `webforge verify init`
- `npm run smoke:onboarding`
- `npm run test:coverage`

结论：

这套 contract 现在不只是“写出来了”，而是有最小可执行验证链。

## 3. 当前可用能力

从使用价值看，当前 WebForge 已经可以承担这些工作：

### 3.1 给业务项目提供 harness 骨架

现在可以初始化一个新项目，并立刻得到：

- `.webforge/` workspace contract
- onboarding protocol
- recovery / observation CLI
- `superpowers` 结果回写入口

### 3.2 给仓库内 agent 提供稳定恢复协议

现在 `Codex / Claude Code` 进入仓库后，已经可以按这条协议工作：

1. `onboard --json`
2. 读 `shouldRead`
3. 看 `recommendedActions`
4. 必要时看 `runtimeLogCommand`
5. 再决定规划、执行或恢复

### 3.3 让 superpowers 结果变成仓库内事实

现在可以把：

- `brainstorming`
- `writing-plans`
- `subagent-driven-development`
- `strategic-compact`

这些 workflow 的结果，通过 `.webforge/superpowers-runs.json`、`.webforge/threads.json` 和 `workflowContext` 接回仓库内状态。

### 3.4 给框架维护者提供最小稳定验证链

现在维护者已经有这条最低验证集：

```bash
npm run build
npm run smoke:onboarding
npm run test:coverage
```

以及这条更聚焦的模板验证入口：

```bash
webforge verify init
webforge verify init --json
```

## 4. 还未完全到位的部分

虽然当前已经满足“可作为最佳实践骨架长期使用”的标准，但还没有到终局版。

剩余缺口主要有这些：

### 4.1 Tool surface 仍然偏轻

当前已经有 contract、state、observation、recovery，但更宽的 agent tool surface 还不够丰富。

更准确地说：

- 现在更像一个稳定 harness skeleton
- 还不是一个非常完整的通用工具生态

### 4.2 更强的权限治理仍然偏轻量

当前已经有 permission profile 和基础 enforcement，但还没有形成更细粒度、更体系化的权限模型。

### 4.3 并行协作依赖 superpowers 工作流，而不是 WebForge 原生执行面

这个方向本身没有问题，因为你已经明确选择：

- `subagent` 隔离交给 `superpowers`
- `worktree` 隔离交给 `superpowers`
- `compact` 也优先复用 `superpowers`

但这意味着 WebForge 后续还需要继续把这些结果的“恢复性”和“可观察性”做得更硬。

### 4.4 维护文档还需要继续沉淀

虽然当前已有：

- README
- operations
- command-reference
- scenario-playbooks
- milestone

但后续如果 contract 再长，仍然需要继续保持文档分层，不然很容易重新塌成“大手册”。

## 6. 当前判断

当前 WebForge 的定位应该明确写成：

> 一个已经达成主线目标的 `v0.1` repo-side harness。

这句话的含义是：

- 它已经符合“按 `learn-claude-code` 的方向，结合 `superpowers`，构建自己的 harness engineering 最佳实践”这个目标
- 它已经可以被真实项目使用
- 它已经有稳定验证链
- 但它还不是最终版，也不应该被误写成“完整通用 agent 平台”

## 7. 下一阶段建议

下一阶段最值的方向，不是推倒重来，而是继续补“薄弱但关键”的几层。

### 建议 1：继续强化 observation surface

重点不是再加更多命令，而是继续让：

- drift
- permission
- workflow recovery
- mailbox / review / checkpoint

这些信号在同一套观察语义里更一致。

### 建议 2：继续强化 permissions

当前有基础 profile，但还可以继续往：

- 更清晰的 action boundary
- 更明确的 approval contract
- 更细粒度的执行限制

这几个方向推进。

### 建议 3：继续把 superpowers 结果的恢复链做硬

后续重点应放在：

- artifact 存在性
- worktree 可恢复性
- compact handoff 的连续性
- thread linkage 的稳定消费路径

### 建议 4：继续保持文档分层

如果后续再加能力，优先考虑：

- 场景手册写“怎么做”
- 命令手册写“怎么调”
- 里程碑文档写“做到哪”

不要再把所有内容重新塞回一份大文档里。

## 8. 结论

这次里程碑的结果可以概括成一句话：

> WebForge 已经从“方向正确的草图”进入“可真实使用、可验证、可恢复的 v0.1 harness 骨架”阶段。

对内部工程来说，这已经不是探索状态，而是一个可以持续迭代的稳定起点。
